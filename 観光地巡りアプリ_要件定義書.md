# 観光地巡りアプリ 要件定義書

## 1. プロジェクト概要

AIと対話しながら観光地を探し、そのままルートナビゲーションまで行えるWebアプリケーション（SPA: シングルページアプリケーション）を開発する。

- **形態**: ブラウザ上で動くSPA
- **フロントエンド方針**: オープンソースのチャットUIをFork（複製）し、そこに地図・しおりパネルを追加改造する
- **開発期間**: 2ヶ月（8週間）
- **開発者**: Pythonエンジニア1年目（学習しながらの開発を前提とする）
- **デプロイ先**: AWS

## 2. 目的

特定地域（山口県・広島県など）を観光する県外の人が、知らない土地でも迷わず、自分の興味関心に合う観光地を周遊できるようにする。

有名な観光スポットを調べたい時に、他の観光地との位置関係、現在地からのルート、移動手段、多少遠回りしてでも寄りたい場所などを、旅のしおりのような形でまとめて提示する。Geminiを搭載しているため、指定した観光地だけでなく、周辺の関連する観光地もレコメンドしてくれる。

ユーザーが行きたい場所とAIとの対話を通じていくつかの観光地をマーキングし、それらを徒歩・車・公共交通機関で周遊するルートを算出、地図上でも確認できるようにする。

## 3. 想定ユーザー

- 観光地（山口県・広島県などの特定地域）を旅したい県外の人

## 4. 機能要件

### 4.1 入力（AIとの対話で収集する項目）

ユーザーがすべてを最初に入力するのではなく、**AIとの対話を通じて段階的に埋めていく**方式とする。

| 項目 | 内容 | 備考 |
|---|---|---|
| 観光地・観光スポット | 行きたいエリアやスポット（例：山口県、広島県） | 必須 |
| 現在地 | ルート計算の起点となる現在位置 | 必須（ルート算出に必要） |
| 箇所数 | 何ヶ所回るか | 必須 |
| 所要時間 | 何時間で回るか | 必須 |
| 移動手段 | 徒歩・車・公共交通機関 | 「多少遠回りしてもよいか」も確認する |
| 観光ジャンル・好み | 歴史、グルメなど | 例：歴史巡り、食べ歩き |
| ルート | 上記条件から生成される最終的な巡回順 | AIとバックエンドが生成 |

**対話フローの例**：
1. ユーザー「京都で半日観光したい」
2. AI「テーマはありますか？世界遺産巡りとか、食べ歩きとか」
3. ユーザー「紅葉を見ながら食べ歩きしたい」
4. AI「移動手段は電車か車、どちらにしますか？多少遠回りしても大丈夫ですか？」
5. 全項目が揃ったら、ルート生成を実行

### 4.2 処理・振る舞い（AIとバックエンドの役割分担）

入力を受け取った後の処理は、以下の役割分担で行う。

1. **Geminiが判断する**：会話の内容から「観光地検索」「ルート計算」など、どの情報の取得が必要かを判断する（実際のデータ取得は行わない）
2. **バックエンドが実行する**：Geminiの判断に基づき、バックエンド（FastAPI）が実際にDB検索・地図API呼び出しなどを実行する
3. **バックエンドがGeminiに結果を戻す**：取得した観光地情報・移動時間などのデータをGeminiに渡す
4. **Geminiが返答を生成する**：渡された結果をもとに、自然な文章のアドバイス・レコメンドを作成する
5. **地図・しおりへの反映**：算出されたルートを地図上にマーキングし、しおり形式の表示と合わせて画面に返す

※ 「Geminiが自分でDBや地図APIにアクセスする」わけではない点に注意。判断はGemini、実行はバックエンド、という主語の違いを設計上明確にする。

### 4.3 出力

以下を1画面にまとめて表示する。

- **地図（ルート表示）**: 訪問順に番号付きピンで表示し、経路を可視化する
- **タイムライン形式のしおり（テキスト）**: 各スポットの訪問時間・説明・提案理由
- **Geminiからのアドバイス・レコメンド**: 指定した観光地以外に関連する観光地の提案など
- **写真**: 各スポットのイメージ（Places API等から取得）

### 4.4 ルート変更機能

- しおりの各スポットに対し、「このスポットを別の場所に変更したい」という対話的な差し替えができること
- 差し替え後、ルート・地図を再計算して再表示する

## 5. 非機能要件

### 5.1 プラットフォーム

- Webアプリ（SPA）のみ。モバイルアプリ化は今回のスコープ外
- 対応ブラウザ：Chromeを想定
- 想定同時利用人数：1人（開発者自身の検証利用を想定）

### 5.2 ログ・データ保持方針

- 利用ログを残すこと自体は必要と考えているが、**目的（デバッグ用か、過去のしおりを見返すためか）と保持期間は今後決める**
- 上記が決まり次第、本ドキュメントを更新する

### 5.3 技術スタック（暫定）

| レイヤー | 技術 | 理由 |
|---|---|---|
| フロントエンド | オープンソースチャットUIをFork（軽量なReact/Vite製を想定） + 自作の地図・しおりパネル | ゼロから作らず開発期間を短縮するため |
| バックエンド | Python / FastAPI | 開発者がPythonエンジニアであるため |
| AI対話 | Gemini API（Function Calling） | 継続的な無料枠があり試作コストを抑えられるため |
| 地図・ルート | 地図API（Google Maps Platform を軸に検討） | Gemini(Google)とのエコシステム親和性 |
| ローカル開発環境 | Docker Compose | フロント・バックエンドを分離してローカル検証するため |
| デプロイ先 | AWS（App Runner を軸に検討） | 設定項目が少なく、1年目エンジニアでも運用しやすいため |

※ 技術選定は2週目のwalking skeleton検証を経て正式に確定する。

## 6. システム構成（概要）

```
SPA（ブラウザ）
 ├─ チャットUI（OSSをFork・改造）
 └─ 地図・しおりパネル（自作）
        │
        ▼
バックエンドサーバー（FastAPI）
 ├─ Gemini API 連携（対話・Function Calling）
 └─ 地図API 連携（ルート計算）
```

## 7. 画面構成（SPA・1画面完結）

- チャット欄（左または上部）：AIとの対話
- 地図・しおりパネル（右または下部）：地図＋タイムライン形式のしおり
- 各しおり項目に「別の候補に変更する」ボタンを設置

## 8. 外部連携

| 連携先 | 用途 |
|---|---|
| Gemini API | 対話処理・Function Callingによる観光地検索の呼び出し判断 |
| 観光地データ（Places API等） | 観光地の情報・写真・レビュー取得 |
| 地図API（Google Maps Platform等） | ルート計算・地図表示 |

## 10. 制約条件

- **開発期間**：2ヶ月（8週間）
- **担当者**：Pythonエンジニア1年目（学習しながらの開発を前提とする）
- **使用技術（指定）**：Python、AWS、Docker、Gemini、BetterChatGPT（Fork元）、SPA

## 11. スコープ外（今回はやらないこと）

- モバイルアプリ化（iOS/Android）
- 複数ユーザー管理（今回は1人利用を想定）
- 認証・ログイン機能
- 複数人でのルート共有機能
- オフライン対応

## 12. 用語集

| 用語 | 意味 |
|---|---|
| SPA | シングルページアプリケーション。画面遷移せず1ページで完結するWebアプリ |
| Function Calling（Tool use） | AIモデルが外部の関数・APIを呼び出す仕組み |
| Fork | 他人のリポジトリを自分のアカウントにまるごとコピーすること |
| Walking Skeleton | 機能は最小限だが、全体の処理の流れ（フロント→バックエンド→外部API→表示）を最初に一本通しておく初期実装 |
